# Документация проекта

Документация подготовлена для ветки `master`, коммит `4f90bdc5a89ef6968308f7df05b7f84ae5eba4d1`.

Цель документа: быстро объяснить, как устроен проект, куда добавлять новые улучшения и какие правила соблюдать, чтобы изменения не ломали импорт отчётов, расчёты и интерфейс.

## 1. Назначение проекта

Проект — локальный Streamlit-дашборд для портфеля ИИС-3. Он импортирует HTML-отчёты брокера Сбербанк, сохраняет исторические снимки портфеля в SQLite и показывает аналитику по позициям, облигациям, денежным потокам, ребалансировке, сделкам и FIRE-сценариям.

Основные возможности:

- загрузка HTML-отчёта брокера через сайдбар Streamlit;
- парсинг отчёта в структурированные модели Python;
- сохранение отчётов, позиций, сделок, денежных потоков и справочных данных в SQLite;
- расчёт стоимости портфеля, P&L, НКД, концентрации и распределения по типам активов;
- расчёт облигационных метрик: YTM, срок до погашения, купонный поток, лестница погашений, рейтинги и качество данных;
- ручные справочники эмитентов, рейтингов, целей ребалансировки и cost basis;
- сценарии принятия решений: что можно рассмотреть к покупке и что можно рассмотреть к сокращению;
- FIRE-блок для долгосрочных финансовых сценариев.

## 2. Быстрый старт

Требования: Python 3.10+ рекомендуется, так как код использует modern type hints вроде `float | None`.

```bash
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate
pip install -r requirements.txt
streamlit run app.py
```

Минимальные зависимости проекта находятся в `requirements.txt`:

```txt
streamlit>=1.30.0
pandas>=2.0.0
plotly>=5.18.0
beautifulsoup4>=4.12.0
```

После запуска откройте приложение Streamlit, загрузите HTML-отчёт брокера в сайдбаре и выберите дату отчёта.

## 3. Основной поток данных

```text
HTML-отчёт брокера
        │
        ▼
parser.py → BrokerReport / Position / CashFlow / Trade / Deposit
        │
        ▼
db.py → SQLite portfolio.db
        │
        ▼
app.py загружает выбранный отчёт
        │
        ├─ moex_api.py: YTM, эмитенты, купоны, погашения, дивиденды, IMOEX
        ├─ concentration.py: доли позиций, эмитенты, HHI, предупреждения
        ├─ analytics/*: облигации, cashflow, рейтинги, качество данных, сценарии
        ├─ rebalancing.py: текущее и целевое распределение
        ├─ portfolio_metrics.py / portfolio_tables.py: метрики и таблицы портфеля
        └─ fire_metrics.py: FIRE-расчёты
        │
        ▼
Streamlit UI: вкладки, метрики, таблицы и графики
```

## 5. Доменная модель

### Типы активов

В проекте используются строковые коды типов активов:

- `bond_ofz_pd` — ОФЗ-ПД;
- `bond_ofz_in` — ОФЗ-ИН;
- `bond_corp` — корпоративные облигации;
- `etf` — БПИФы / ETF;
- `stock` — акции.

Облигационные типы собраны в `concentration.BOND_ASSET_TYPES`:

```python
BOND_ASSET_TYPES = ("bond_ofz_pd", "bond_ofz_in", "bond_corp")
```

### Основные сущности отчёта

`parser.py` создаёт следующие dataclass-модели:

- `Position` — позиция в портфеле;
- `CashFlow` — движение денежных средств;
- `Trade` — сделка;
- `Deposit` — пополнение ИИС;
- `SecurityInfo` — справочная информация по бумаге;
- `BrokerReport` — полный отчёт с периодом, инвестором, суммами и списками сущностей.

### Основные таблицы SQLite

`db.py` создаёт и использует таблицы:

- `reports` — импортированные отчёты;
- `positions` — позиции отчёта;
- `cash_flows` — движения денежных средств;
- `trades` — сделки;
- `deposits` — пополнения;
- `coupon_calendar` — купонный календарь;
- `dividend_calendar` — дивидендный календарь;
- `rebalance_targets` — целевые доли ребалансировки;
- `cost_basis` — средняя цена покупки;
- `bond_maturities` — погашения облигаций;
- `bond_amortizations` — амортизации облигаций;
- `fire_assets` — внешние активы FIRE-блока;
- `data_sync_status` — статусы синхронизации внешних данных;
- `issuer_reference` — ручной справочник эмитентов;
- `bond_ratings` — ручной справочник рейтингов.

## 6. Правила расширения проекта

### 6.1. Добавление нового поля из HTML-отчёта

1. Найдите нужную таблицу или раздел отчёта в `parser.py`.
2. Добавьте поле в соответствующий dataclass.
3. Обновите `parse_report()` так, чтобы поле заполнялось из HTML.
4. Обновите `db.py`:
   - добавьте поле в схему новой установки;
   - добавьте миграцию в `SCHEMA_MIGRATIONS` для существующей базы;
   - обновите `import_report()` и функции чтения.
5. Добавьте отображение в `app.py` или `portfolio_tables.py`.
6. Добавьте тест на парсинг с HTML fixture.

### 6.2. Добавление новой аналитической метрики

Рекомендуемый путь:

1. Создайте чистую функцию в отдельном модуле или в подходящем `analytics/*.py`.
2. Передавайте на вход обычные структуры: `list[dict]`, `Mapping`, `DataFrame`, дату `as_of_date`.
3. Не обращайтесь к Streamlit внутри аналитической функции.
4. Возвращайте словарь или список словарей, удобный для таблиц и графиков.
5. Подключите функцию в `app.py`.
6. Добавьте unit-тест на чистую функцию.

Пример шаблона:

```python
def build_new_metric(
    positions: list[dict],
    *,
    as_of_date: date,
    threshold: float = 0.10,
) -> dict[str, Any]:
    # 1. нормализовать входы
    # 2. посчитать rows / summary
    # 3. вернуть serializable dict
    return {"rows": [], "summary": {}}
```

### 6.3. Добавление новой вкладки Streamlit

1. Добавьте название вкладки в список `st.tabs([...])` в `app.py`.
2. Создайте блок `with tab_new:`.
3. Вынесите тяжёлые расчёты из UI в отдельную функцию или модуль.
4. Для таблиц используйте подготовленные `DataFrame`, а форматирование держите ближе к UI.
5. Если вкладка использует внешние данные, добавьте caption о свежести данных через `db.get_data_sync_freshness()`.

### 6.4. Добавление внешнего источника данных

1. Создайте отдельный модуль клиента или расширьте `moex_api.py`, если источник относится к MOEX.
2. Добавьте retry/backoff и понятную обработку ошибок.
3. Не блокируйте основной UI ошибкой внешнего API: возвращайте пустые данные и сохраняйте статус ошибки.
4. Записывайте статус в `data_sync_status`, если данные синхронизируются по ISIN или сущности.
5. Для тестов мокайте сетевой слой.

### 6.5. Добавление таблицы или поля в SQLite

1. Увеличьте номер версии в `SCHEMA_MIGRATIONS`.
2. Добавьте SQL-миграцию.
3. Обновите базовую схему в `init_db()` для чистой установки.
4. Добавьте функции чтения/записи рядом с похожими CRUD-функциями.
5. Проверьте сценарий: новая база и существующая база.

Пример:

```python
(
    5,
    "add_new_table",
    """
    CREATE TABLE IF NOT EXISTS new_table (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        key TEXT NOT NULL UNIQUE,
        value TEXT DEFAULT '',
        updated_at TEXT NOT NULL DEFAULT (datetime('now'))
    );
    """,
),
```

### 6.6. Добавление нового decision-сценария

1. Размещайте правила в `analytics/decision_scenarios.py`.
2. Делайте функцию чистой: входы через параметры, выход — словарь с `candidates`, `excluded_summary`, `weights` или похожей структурой.
3. Возвращайте причины решения в машиночитаемом виде и человекочитаемом виде.
4. Добавьте настройки в UI рядом с уже существующими сценариями.
5. Покройте сортировку, исключения и отключение факторов unit-тестами.

## 7. Текущие сценарии принятия решений

### `build_buy_candidates()`

Формирует список облигаций, которые можно рассмотреть к покупке. Основные входы:

- свободные деньги;
- YTM по ISIN;
- даты погашения;
- доли позиции и эмитента;
- текущие и целевые доли типов активов;
- лимиты концентрации;
- минимальная YTM;
- максимальный срок до погашения;
- флаги исключения бумаг без YTM или даты погашения.

Выход:

- `candidates` — отсортированные кандидаты;
- `excluded_summary` — причины исключения;
- `total_positions_seen`;
- `total_bond_positions`.

### `build_reduce_candidates()`

Формирует список позиций-кандидатов на сокращение по прозрачному `risk_score`.

Факторы по умолчанию:

- большая доля позиции;
- большая доля эмитента;
- отсутствие YTM;
- отсутствие даты погашения;
- низкая YTM относительно срока;
- отсутствие рейтинга;
- длинный срок до погашения;
- проблемы качества данных.

Каждый фактор имеет вес, может быть отключён и возвращает вклад в итоговый score.

## 8. Тестирование

Базовый запуск тестов:

```bash
python -m unittest discover -s tests
```

Сейчас в `tests/test_decision_scenarios.py` покрыты:

- сортировка кандидатов на покупку по YTM и доле позиции;
- причины исключения кандидатов;
- сценарий допуска бумаги без YTM с предупреждением;
- сортировка кандидатов на сокращение по `risk_score`;
- отключение фактора риска.

Рекомендуемые следующие тесты:

- parser fixture-тесты на HTML-отчётах;
- тесты миграций SQLite на временной базе;
- тесты MOEX-клиента с моками сетевых ответов;
- тесты расчётов `concentration.py`, `analytics/bonds.py`, `analytics/cashflows.py`, `analytics/data_quality.py`;
- smoke-тесты подготовки таблиц для Streamlit.

## 11. Принципы разработки

- Парсинг, хранение, расчёты и UI должны оставаться разделёнными.
- Новые расчёты лучше добавлять как чистые функции и тестировать без Streamlit.
- Любая новая таблица или поле должны поддерживать как новую установку, так и существующую базу через миграцию.
- Внешние API не должны ломать работу приложения при временной недоступности.
- Отсутствие данных должно отображаться явно: `нет данных`, `—`, предупреждение или строка в отчёте качества.
- Финансовые сценарии должны оставаться рекомендациями для проверки, а не автоматическими торговыми действиями.
